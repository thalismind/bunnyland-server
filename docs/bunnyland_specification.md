# bunnyland Specification

**Status:** working draft 0.1
**Primary runtime:** async Python (single process, multi-threading and async tasks as needed)
**Primary state model:** Relics ECS database (https://github.com/ssube/relics)
**Primary interface:** Discord (https://discordpy.readthedocs.io/en/stable/)
**Primary design goal:** an asynchronous social sandbox where human players and LLM agents share persistent characters in an emergent life/colony/survival simulation.

---

## 1. Core Requirements

1. bunnyland uses the **Relics ECS database** as the authoritative in-process world-state engine.
2. The application is written in **async Python** around a single Relics world actor.
3. Components, commands, events, plugin declarations, and structured LLM payloads use **Pydantic models or dataclasses** for validation and type safety.
4. The internal architecture is **event-driven**, with typed events, an event bus, volatile command queues, and a Command-pattern-style command/action layer.
5. LLM-driven agents are represented as persistent ECS character entities, controlled by swappable controller entities.
6. ECS components and relationships are converted into prompt fragments for LLM agents and human players.
7. Each character has private long-term notes and memory, backed by a vector database such as ChromaDB.
8. Gameplay is based on **Action Points** and **Focus Points**, both of which regenerate over real time.
9. Prompts are generated for each agent or player based on character state, inventory, feelings, room state, recent events, notes, and available commands.
10. The world is generated by an LLM DM/world-builder agent from an initial seed or prompt.
11. The world is a graph of room entities, with directed connections between them representing possible paths.
12. The game is asynchronous and has no fixed global turn order.
13. Characters may interact through dialogue, exploration, item use, survival mechanics, crafting, combat, colony work, family/life simulation, and plugin-defined mechanics.
14. The game is extensible through plugins that may define ECS mechanics, controllers, generators, projections, integrations, policies, commands, events, prompt parts, or content.
15. Actions are implemented as command handlers and can be exposed to LLM agents as tool calls.
16. Human players and LLM agents use the same command/action surface and receive the same underlying prompt context.
17. The game is social and primarily played through Discord.

---

## 2. Abstract

bunnyland is an asynchronous social sandbox where humans and LLMs control persistent characters in a shared world. Characters exist independently of their current controller. A human Discord player may take over a character from an LLM, release the character back to an LLM, or suspend the character with a no-op controller.

The game is built on top of Relics ECS. Relics stores the authoritative current state of rooms, characters, items, needs, relationships, controllers, and world objects. Components describe state; relationships describe graph structure; systems advance simulation; commands express attempted actions; typed events describe what happened.

The core gameplay loop is asynchronous. Players and LLMs may submit commands at any time. Commands run on the next world tick if the character has enough Action and/or Focus points. If the character does not have enough points, the command may be denied immediately or queued in volatile engine memory according to the submitter's preference. Queued commands do not survive a server restart.

The simulation emphasizes emergent behavior inspired by The Sims, RimWorld, and Conan Exiles: needs, moods, environment, relationships, colony work, survival pressure, and world interaction all feed into each other. The system deliberately avoids monolithic mechanics. Hunger and thirst are separate components and systems. Sleep, hygiene, affect, social needs, inventory, weather, injury, crafting, combat, family life, and other mechanics are likewise separate.

LLM agents are not the physics engine. They produce commands/tool calls against the same validated action surface available to humans. The ECS, command handlers, systems, policies, and validators decide what actually changes.

The world is generated by an LLM DM/world-builder. The world-builder proposes structured content: rooms, exits, objects, NPCs, initial components, and story hooks. The engine validates the proposal and instantiates entities, components, and relationships. The world-builder does not directly mutate the Relics world.

Each character has a private notebook and memory collection. Taking notes and remembering/searching notes spend Focus. Notes are private by default and are not discoverable physical objects. Writing on paper, books, signs, or other world items is a separate physical writing mechanic.

## 3. Architectural Boundaries

### 3.1 Relics is the authoritative world state

Relics stores current ECS state:

```text
World
  Entities
  Components
  Relationships / Edges
  Systems
  Observers
  Custom Events
  Indexes
  Persistence snapshots
```

Relics is used for:

```text
current room graph
characters and items
controllers as entities
components for needs, points, affect, inventory, environment, policy, etc.
relationships such as Contains, ExitTo, ControlledBy, Holding, Wearing
simulation systems
query/index support
snapshots
```

Bunnyland's shared graph-query layer compiles connected conjunctive component and edge
terms to these Relics primitives. Queries are bounded to eight terms, six variables, 100
rows, and 10,000 candidate expansions. They reject unknown types, missing fixed entities,
undefined output variables, disconnected products, and exhausted budgets. Trusted scripts
may use graph selectors; agents receive only typed claim-scoped questions registered by
plugins, never raw graph specifications or unrestricted ECS discovery.

Relics is **not** the whole backend. bunnyland wraps Relics with:

```text
async world actor
volatile command queues
typed event bus
event log
command audit log
Discord integration
LLM orchestration
memory/vector services
room-summary projections
plugin loading
```

### 3.2 Single world actor

The Relics `World` is owned by a single world actor. Discord handlers, LLM workers, timers, and external services do not mutate ECS state directly.

All world mutations flow through:

```text
submitted command
→ volatile command queue
→ world actor tick
→ validation
→ command handler
→ ECS mutation
→ typed events
→ projections / side effects
```

This prevents race conditions such as:

```text
two characters taking the same item
late LLM response overriding a human takeover
hunger/thirst systems mutating while a command handler mutates the same character
Discord button presses applying stale state
```

### 3.3 LLMs propose, systems decide

LLMs can:

```text
choose commands
speak in character
write private notes
search memories
propose world content
propose ambiguous action intent
produce reflections
narrate flavor text where allowed
```

LLMs cannot:

```text
directly mutate ECS state
create pregnancy/death/ownership/relationship state through prose alone
ignore Action/Focus costs
bypass controller generation checks
bypass boundaries or world policy
bypass reachability, containment, or inventory rules
```

## 4. Type Safety and Mutation Style

### 4.1 Models

Use either Pydantic models or dataclasses for:

```text
components
relationships / edges
commands
events
plugin declarations
worldgen proposals
prompt context models
memory entries
configuration
```

Pydantic is preferred for external or serialized boundaries. Dataclasses are acceptable for simple internal ECS components when Relics supports them cleanly.

### 4.2 Prefer replacement over in-place mutation

Components should generally be treated as immutable values. Prefer replacing a component rather than mutating internal lists/sets/dicts in place.

Preferred:

```python
new_component = replace(old_component, current=old_component.current + 1)
entity.add_component(new_component)
```

Avoid:

```python
component.allowed.add(tag)
```

This matters for:

```text
observers
indexes
persistence
projection invalidation
debugging
test determinism
```

Use immutable containers where possible:

```text
frozenset instead of set
tuple instead of list
frozen Pydantic/dataclass models where appropriate
```

## 5. Runtime Tick Model

### 5.1 Commands can be submitted at any time

Players and LLM controllers may submit commands whenever they want.

On the next world tick:

```text
if affordable and valid → command runs
if not affordable → command is denied or queued, based on submitter preference
if queued → command waits in volatile memory until affordable, invalid, cancelled, expired, or server restart
```

Queued commands do **not** survive a server restart.

Command queues are engine memory, not ECS state and not durable persistence.

### 5.2 No persistent Decision Session component

There is no required `DecisionSessionComponent`.

Input readiness is derived from:

```text
character state
controller state
ActionPointsComponent
FocusPointsComponent
SuspendedComponent / DownedComponent / DeadComponent
command costs
policy checks
```

The engine may maintain transient readiness maps, prompt leases, Discord message IDs, or queued command state, but these are not authoritative ECS world state.

### 5.3 Two command lanes

Each active character has two volatile command lanes.

#### Focus lane

Focus-only private commands:

```text
take note
remember / check notes / search notes
future private reflection/planning tools
```

These spend Focus but do not alter the shared physical world.

#### World lane

World-facing commands:

```text
move
take
drop
put
use
eat
drink
say
tell
sleep
wake
write on physical item
combat/crafting/plugin actions
```

These may spend Action, Focus, or both.

Dialogue is world-lane because it affects the shared social world.

```text
say/tell cost Action + Focus
```

### 5.4 Queue semantics

Each lane is FIFO per character.

A queued command is an intent, not a point reservation. It is revalidated when it is about to execute.

Reject or cancel a queued command if:

```text
controller generation changed
character is no longer controlled by that controller
character became dead
character became suspended and the command requires active state
target no longer exists
target is no longer reachable
target was taken/consumed/changed by someone else
policy/boundary no longer allows it
command expired
command payload is invalid
```

When a character changes controller or becomes suspended, queued commands from the previous controller are invalidated by controller generation checks and may be flushed by the engine.

### 5.5 Initiative

Characters have initiative.

```python
@dataclass(frozen=True)
class InitiativeComponent(Component):
    score: float = 0.0
```

Characters with executable queued commands act in descending initiative order.

If initiative scores tie, the tie-break order is randomized each tick.

For MVP:

```text
one world-lane command per character per tick
focus-lane commands may run before that character's world-lane command
```

If later multiple world-action passes are allowed in the same tick, the scheduler should repeat initiative order rather than allowing one high-resource character to spend all Action before others can respond.

### 5.6 Tick phases

The world tick proceeds in deterministic phases:

```text
1. Ingest submitted commands into volatile queues.
2. Advance world clock.
3. Regenerate Action and Focus.
4. Run passive simulation systems.
   - weather
   - temperature
   - moisture
   - light
   - hunger
   - thirst
   - sleep
   - injury/status
   - downed/recovery checks
5. Process queued commands in initiative order.
   - Focus lane
   - World lane
6. Run consequence systems.
   - observation
   - thought creation
   - affect updates
   - relationship updates
   - memory writes
   - downed/death transitions
7. Emit/bridge typed events.
8. Update projections.
   - room summary invalidation/rebuild
   - recent context
   - prompt dirtiness
9. Dispatch Discord/LLM notifications.
10. Persist snapshots/events according to policy.
```

Simulation systems run before characters act.

## 6. Action and Focus Points

### 6.1 Components

Approved names:

```python
@dataclass(frozen=True)
class ActionPointsComponent(Component):
    current: float = 0.0
    maximum: float = 5.0
    regen_per_hour: float = 1.0
    overflow_maximum: float | None = None


@dataclass(frozen=True)
class FocusPointsComponent(Component):
    current: float = 0.0
    maximum: float = 3.0
    regen_per_hour: float = 0.5
    overflow_maximum: float | None = None
```

Exact initial values and regeneration rates are tuning data, not architecture. They will be found through gameplay tuning and optimization.

### 6.2 Regeneration

Action and Focus regenerate in real time for every character, regardless of controller.

```text
Discord-controlled characters regenerate.
LLM-controlled characters regenerate.
Suspended/no-op-controlled characters regenerate.
```

Suspended characters do not consume points because their suspended/no-op controller performs no actions, so they naturally become fully recharged.

Action and Focus may regenerate at different rates.

Sleep may increase regeneration.

Injury, exhaustion, illness, severe hunger, severe thirst, stress, downed state, or plugin-defined conditions may decrease regeneration.

Bonuses may allow overflow beyond normal maximum, up to `overflow_maximum` or package-defined limits.

### 6.3 Command costs

```python
@dataclass(frozen=True)
class CommandCost:
    action: int = 0
    focus: int = 0
```

Examples:

```text
move              Action 1, Focus 0
take              Action 1, Focus 0
drop / put        Action 1, Focus 0
use               Action varies, Focus varies
eat / drink       Action 1, Focus 0
say / tell        Action 1, Focus 1
take note         Action 0, Focus 1
remember/search   Action 0, Focus 1
write on paper    Action 1, Focus 1
sleep             Action 1, Focus 0
release to LLM    control command, no point cost
```

A character with only 1 Focus cannot search notes and speak in the same immediate action sequence if both cost 1 Focus.

## 7. Characters, Controllers, and Handoff

### 7.1 Characters are persistent

A character is a persistent ECS entity. It is not the same thing as its controller.

A character may have:

```text
identity
body
needs
Action/Focus points
inventory
held/worn equipment
affect
thoughts
relationships
private notes
memory collection
current room
controller relationship
```

The controller is replaceable.

### 7.2 Controller types

Supported controller types:

```text
Discord player controller
Web player controller
MCP agent controller
LLM agent controller
Behavioral (behavior-tree) controller
Scripted controller
Suspended/no-op controller
```

Behavioral and scripted controllers are engine-driven like the LLM controller — the engine
proposes their actions and validates them on the next tick — but they are deterministic and
make no model call. A behavioral controller ticks a named behavior tree (resolved from a
registry); a scripted controller replays a named sequence of tool calls, optionally looping.

LLM controllers may vary by:

```text
model
temperature
system prompt style
tool policy
cost profile
agent personality
memory settings
```

### 7.3 Controller entities

Controllers can be represented as ECS entities.

```python
@dataclass(frozen=True)
class DiscordControllerComponent(Component):
    discord_user_id: int
    default_channel_id: int
    mention_on_ready: bool = True
    allow_dm: bool = False


@dataclass(frozen=True)
class LLMControllerComponent(Component):
    profile_name: str
    model: str
    temperature: float
    max_tokens: int
    system_style: str = "in_character"
    tool_policy: str = "character_actions"


@dataclass(frozen=True)
class SuspendedControllerComponent(Component):
    reason: str = "offline"
```

Controller assignment:

```python
@dataclass(frozen=True)
class ControlledBy(Edge):
    generation: int
    since_epoch: int
```

Every control change increments `generation`.

Any command, Discord button, or LLM tool response must carry the controller generation it was created under. If the generation no longer matches, reject the command as stale.

This prevents:

```text
LLM prompted
human takes over
late LLM response arrives
late LLM action mutates human-controlled character
```

### 7.4 Controller transitions

Supported transitions:

```text
LLM -> Discord       human takes control
Discord -> LLM       human releases character to LLM
Discord -> Suspended human suspends/sleeps/parks character
Suspended -> Discord human resumes
LLM -> Suspended     admin/system pause or story state
Suspended -> LLM     admin/system resumes autonomous control
```

On transition:

```text
increment controller generation
invalidate old queued commands
cancel stale prompts/LLM requests
emit typed controller event
regenerate prompt if needed
```

### 7.5 Human takeover

A human may take over an eligible character according to control policy.

A player-created character normally defaults to owner-only control.

Generated/unclaimed characters may be configured as:

```text
owner_only
party_only
any_human
admin_only
unclaimed_only
```

### 7.6 Release to LLM

Releasing a character to the LLM keeps the character active in the world.

Consequences:

```text
LLM can spend Action/Focus.
LLM can make decisions.
LLM-controlled active characters can die.
The character remains part of the simulation.
```

The LLM should be told that it is resuming autonomy from the character's current state and memories, not from the human player's private out-of-character thoughts.

### 7.7 Suspended/no-op controller

Suspending a character assigns a no-op controller and adds `SuspendedComponent`.

Suspended characters:

```text
regenerate Action and Focus
do not spend Action or Focus
do not receive autonomous LLM prompts
are ignored by most harmful/world-participation systems
cannot die
cannot be targeted by PvP
cannot start pregnancy/family actions
```

## 8. Active, Suspended, Downed, and Dead States

### 8.1 Suspended component

Suspension is indexable via marker component.

```python
@dataclass(frozen=True)
class SuspendedComponent(Component):
    reason: str = "offline"
    suspended_at_epoch: int = 0
```

Each system defines its own query. Most systems that can harm or involve characters in the world filter out suspended characters.

Examples:

```text
Action/Focus regeneration includes suspended characters.
Hunger/thirst damage usually excludes suspended characters.
Death/downed systems exclude suspended characters.
LLM controller dispatch excludes suspended characters.
Prompting excludes suspended characters.
Pregnancy beginning excludes suspended characters.
```

### 8.2 Active definition

A character is active if:

```text
it has CharacterComponent
it does not have SuspendedComponent
it does not have DeadComponent
it is controlled by a Discord controller or LLM controller
```

### 8.3 Death rule

Characters can die, but only while active.

```text
LLM-controlled active characters can die.
Discord-controlled active characters can die, even if the human is idle.
Suspended characters cannot die.
```

### 8.4 Downed state

The game should be cooperative. Characters usually become downed before dying.

```python
@dataclass(frozen=True)
class DownedComponent(Component):
    downed_at_epoch: int
    cause: str
    checks_remaining: int = 3
    stable: bool = False


@dataclass(frozen=True)
class DeadComponent(Component):
    died_at_epoch: int
    cause: str
    source_event_id: str | None = None
```

Downed characters may revive through:

```text
successful recovery checks
help from another character
medicine
magic
food/water/rest
plugin-defined mechanics
```

Death occurs after failed recovery, explicitly lethal events, or package-defined consequences.

## 9. ECS World Model

### 9.1 Entities

Anything meaningful can be an entity:

```text
world singleton
rooms
characters
controllers
items
containers
furniture
resources
notes
thoughts
goals
jobs
factions
conversations
generated-content proposals
```

### 9.2 Components describe state

Components should be small, typed, and focused.

Good:

```text
HungerComponent
ThirstComponent
SleepNeedComponent
AffectComponent
ContainerComponent
FoodComponent
DrinkableComponent
SuspendedComponent
```

Bad:

```text
NeedsComponent
EnvironmentComponent
CharacterStateComponent
EverythingComponent
```

### 9.3 Systems own logic

Systems define their own queries.

Most harmful/world-participation systems filter out `SuspendedComponent`.

Each system should generally:

```text
read broadly
write narrowly
emit typed events
```

Example:

```text
ThirstSystem reads Thirst, Temperature, Moisture, Body, Sleep.
ThirstSystem writes Thirst.
ThirstSystem emits ThirstChangedEvent / DehydrationRiskEvent.
```

Avoid systems that directly mutate many unrelated components.

### 9.4 Each mechanic is separate

A mechanic gets its own component/system/action/event/prompt set when it has distinct:

```text
causes
actions
consequences
prompt language
tests
tuning
```

Hunger and thirst are separate because they are fed in different ways and affected by different environmental conditions.

## 10. Containment, Inventory, and Room Graph

### 10.1 Contains is canonical

Use one physical containment relationship: `Contains`.

No separate `Carrying` edge.

```text
Room contains character.
Room contains chest.
Chest contains apple.
Character contains backpack.
Backpack contains note.
Character contains sword.
Character is holding sword.
Character is wearing armor.
```

`Contains` answers where something physically is.

```python
class ContainmentMode(StrEnum):
    ROOM_CONTENT = "room_content"
    INVENTORY = "inventory"
    CONTAINER = "container"
    SURFACE = "surface"
    ATTACHED = "attached"


@dataclass(frozen=True)
class Contains(Edge):
    mode: ContainmentMode = ContainmentMode.CONTAINER
    visible: bool = True
    discovered: bool = True
    order: int = 0
```

### 10.2 Holding and Wearing are overlays

Held and worn items must also be contained by the character.

```python
@dataclass(frozen=True)
class Holding(Edge):
    slot: str = "hand"


@dataclass(frozen=True)
class Wearing(Edge):
    slot: str
```

Valid:

```text
Juniper --Contains(mode=inventory)--> dull knife
Juniper --Holding(slot=right_hand)--> dull knife
```

Invalid:

```text
Juniper --Holding--> dull knife
Mosslit Burrow --Contains--> dull knife
```

### 10.3 Containment invariants

```text
Every physical entity has at most one direct Contains parent.
Physical containment cannot cycle.
Rooms can contain characters, objects, furniture, hazards, containers, and resources.
Characters contain inventory.
Containers contain items.
Held items must be contained by the holder.
Worn items must be contained by the wearer.
Taking transfers Contains to the character.
Dropping transfers Contains to the current room.
Putting transfers Contains to the target container.
Moving a container moves everything inside it.
```

### 10.4 Room graph

Rooms are entities.

Connections are directed room-to-room edges.

```python
@dataclass(frozen=True)
class ExitTo(Edge):
    direction: str
    label: str = ""
    locked: bool = False
    hidden: bool = False
    action_cost: int = 1
```

The DM/world-builder generates rooms and exits as structured proposals.

## 14. Speech, Intent, and Social Interpretation

### 14.1 Flirting is not a separate action

Social meaning flows through dialogue.

Do not create separate base verbs for:

```text
flirt
compliment
insult
apologize
threaten
confess
joke
bargain
propose
```

Use:

```text
say
tell
```

Then interpret the speech.

### 14.2 SpeechIntent enum

Start with a reasonable set and evolve during implementation.

```python
class SpeechIntent(StrEnum):
    NEUTRAL = "neutral"
    INFORM = "inform"
    QUESTION = "question"
    REQUEST = "request"
    OFFER = "offer"
    JOKE = "joke"
    INSULT = "insult"
    THREAT = "threat"
    COMFORT = "comfort"
    APOLOGY = "apology"
    PRAISE = "praise"
    FLIRT = "flirt"
    CONFESSION = "confession"
    PROMISE = "promise"
    GOSSIP = "gossip"
```

Author may provide intent.

If no intent is provided, infer one from the text.

Store:

```text
author_intent
inferred_intent
final_interpretation
```

This helps avoid misinterpretation caused by lack of body language, and supports intentionally ridiculous tone declarations.

Example:

```text
say "Your mother was a herring and your father smells of elderberries" intent=insult
```

### 14.3 Speech flow

```text
SayCommand / TellCommand
→ validate Action + Focus
→ create typed speech event
→ determine observers/hearers
→ infer intent if needed
→ apply policy/boundary filters
→ relationship update
→ thought creation
→ memory write
→ prompt dirty events
```

### 14.4 Visibility

MVP rules:

```text
say is heard by active awake characters in the room
tell is directed/private to one character
tell is not overheard in MVP
suspended characters perceive nothing
```

Overhearing, stealth, adjacent-room hearing, and body-language systems can be added later.

## 15. Notes, Memory, and Reflection

### 15.1 Private notes

Each character has a private notebook/memory collection.

Taking notes:

```text
can happen at any time
spends Focus
is private by default
is not physically discoverable
may be submitted from DM, public channel, Discord UI, or LLM tool
```

Focus actions for human players typically happen by DM with the bot, but note-taking can happen anywhere.

Input surface does not change the budget rule.

```text
take note from DM       -> spends Focus
take note from channel  -> spends Focus
take note by LLM tool   -> spends Focus
```

### 15.2 Remember/search

Remembering or searching notes requires Focus.

```text
check recent notes
search notes by query
vector search long-term memory
```

Results are private.

### 15.3 Vector database

Use a vector DB such as ChromaDB for long-term memory, notes, reflections, and retrieval.

Recommended memory collections:

```text
one private collection per character
shared collections as v2 feature
world/system collections for lore and generated content
```

### 15.4 Notes versus physical writing

Private notes are memory entries.

Physical writing is world state.

```text
take note "Hazel distrusts the basin"
  -> private memory, Focus cost, not discoverable

write "Hazel distrusts the basin" on paper
  -> physical item mutation, Action + Focus, discoverable
```

Shared notes are a v2 feature.

### 15.5 Reflections

Reflections are later-stage memory mechanics.

A reflection can summarize recent events or derive character goals, but it should create subjective character memory/affect/goals, not rewrite objective world facts.

## 17. Room Summaries and Projections

### 17.1 Event-driven, not tick systems

Room summary updates do not need to be a ticking ECS system.

They are event-driven projections/caches.

```text
world event happens
→ affected room summary marked dirty
→ summary rebuilt lazily or by projection worker
→ prompt builder uses latest summary
```

### 17.2 Structured facts and prose summary

Keep deterministic room facts separate from prose summary.

Structured facts drive actions.

Prose summary improves readability.

The summary must never become the source of truth.

### 17.3 Invalidation rules

Dirty room summary when:

```text
Contains changes in or under room
Exit changes
visible entity description changes
container/door open state changes
lock state changes
light band changes
weather/environment band changes
notable room event occurs
character enters or leaves
```

Use semantic bands, not tiny numeric changes.

```text
light: dark / dim / lit / bright
temperature: cold / cool / mild / warm / hot
moisture: dry / damp / wet / flooded
cleanliness: filthy / messy / clean
```

MVP summaries are deterministic/template-based. LLM prose summaries can come later.

## 20. Policy, Boundaries, PvP, and Adult/Life Simulation

### 20.1 Boundary tags

```python
class BoundaryTag(StrEnum):
    FLIRTING = "flirting"
    ROMANCE = "romance"
    ADULT = "adult"
    PREGNANCY = "pregnancy"
    PVP = "pvp"
    LETHAL_PVP = "lethal_pvp"
    THEFT = "theft"
    PICKPOCKETING = "pickpocketing"
```

### 20.2 Character boundary component

```python
@dataclass(frozen=True)
class CharacterBoundaryComponent(Component):
    allowed: frozenset[BoundaryTag] = frozenset()
    denied: frozenset[BoundaryTag] = frozenset()
```

Rules:

```text
World/plugin policy defines defaults.
Characters can opt in or opt out.
Denied always wins.
Admins cannot override character boundaries.
```

Admins may moderate the server/world, but they cannot make a character opt into a boundary tag that the character/player denied.

### 20.3 World policy

World and plugin policy define what is available at all.

```python
@dataclass(frozen=True)
class WorldPolicyComponent(Component):
    enabled: frozenset[BoundaryTag] = frozenset()
    disabled: frozenset[BoundaryTag] = frozenset()
```

A mechanic is allowed only if:

```text
world/plugin policy enables it
character boundaries allow it
no relevant character denies it
action-specific prerequisites pass
```

### 20.4 PvP

PvP is optional and policy-gated.

PvP means player-player combat. It does not include dialogue.

For now:

```text
pickpocketing is disabled
stealing directly from character inventory is disabled
dialogue conflict is not PvP
combat between player-controlled characters is PvP
lethal PvP is a separate tag/policy
```

Theft from containers, ownership disputes, sabotage, and item destruction should be separate mechanics/policies and not silently bundled into PvP.

### 20.5 Romance/adult/family systems

Romance, adult, pregnancy, and family simulation are package-defined life-sim mechanics.

Flirting is not its own action; it is speech intent/interpretation.

Major state transitions require validated commands/events, not freeform narration.

Examples of major transitions:

```text
becoming partners
ending partnership
starting family storyline
pregnancy beginning
birth
parent/child edge creation
```

### 20.6 Pregnancy

Pregnancy usually belongs to a life-sim plugin.

Pregnancy cannot begin while any relevant participant is suspended.

Default suspended progression behavior is option B:

```text
Pregnancy time can progress while suspended.
Major events such as birth are deferred until the character becomes active.
```

If a character logs out while pregnant and the due time passes:

```text
pregnancy can become due
birth does not automatically resolve while suspended
on resume, a BirthDueComponent or pending birth event is processed
```

This avoids both:

```text
pregnancy freezing forever while offline
player returning to find the child already aged significantly
```

Potential components:

```python

    stage: str = "adult"  # child, teen, adult, elder

    can_cause_pregnancy: bool = False
    fertility: float = 1.0
    species_group: str = "bunny"
    pregnancy_blocked: bool = False

    started_at_epoch: int
    due_at_epoch: int
    source_event_id: str | None = None


@dataclass(frozen=True)

    due_since_epoch: int
```

Family edges:

```python

    pass
```

LLM-controlled characters may be given fewer constraints by world/plugin configuration to encourage emergent behavior, but typed policies and validators still decide what can become ECS state.



### 21.2 Plugin model

```python
class EcsContribution(BaseModel):
    model_config = ConfigDict(arbitrary_types_allowed=True, frozen=True)

    components: tuple[type, ...] = ()
    edges: tuple[type, ...] = ()
    systems: tuple[type, ...] = ()
    observers: tuple[type, ...] = ()


class CommandContribution(BaseModel):
    model_config = ConfigDict(arbitrary_types_allowed=True, frozen=True)

    command_types: tuple[type, ...] = ()
    action_handlers: tuple[type, ...] = ()
    action_definitions: tuple[Any, ...] = ()
    typed_events: tuple[type, ...] = ()


class RuntimeContribution(BaseModel):
    model_config = ConfigDict(arbitrary_types_allowed=True, frozen=True)

    controller_factories: tuple[Any, ...] = ()
    generator_factories: tuple[Any, ...] = ()
    service_factories: tuple[Any, ...] = ()
    projection_factories: tuple[Any, ...] = ()
    integration_factories: tuple[Any, ...] = ()


class ContentContribution(BaseModel):
    model_config = ConfigDict(arbitrary_types_allowed=True, frozen=True)

    prefabs: tuple[Any, ...] = ()
    prompt_parts: tuple[Any, ...] = ()
    generation_capabilities: tuple[str, ...] = ()
    intent_normalizers: tuple[Any, ...] = ()
    generation_enrichers: tuple[Any, ...] = ()


class PolicyContribution(BaseModel):
    model_config = ConfigDict(arbitrary_types_allowed=True, frozen=True)

    boundary_tags: frozenset[str] = Field(default_factory=frozenset)
    world_defaults: dict[str, Any] = Field(default_factory=dict)
    config_schema: type | None = None


class Plugin(BaseModel):
    model_config = ConfigDict(arbitrary_types_allowed=True, frozen=True)

    id: str
    name: str
    version: str = "0.1.0"

    dependencies: tuple[str, ...] = ()
    default_enabled: bool = True

    ecs: EcsContribution = Field(default_factory=EcsContribution)
    commands: CommandContribution = Field(default_factory=CommandContribution)
    runtime: RuntimeContribution = Field(default_factory=RuntimeContribution)
    content: ContentContribution = Field(default_factory=ContentContribution)
    policy: PolicyContribution = Field(default_factory=PolicyContribution)
```

### 21.3 Plugin loading

The launcher discovers plugins from installed package entry points.

Desired shape:

```text
bunnyland serve \
  --plugin vendor.foo \
  --plugin vendor.bar
```

Semantics:

```text
Installed distributions declare the `bunnyland.plugins` entry-point group.
An entry point returns one or more Plugin objects.
--plugin selects which discovered plugin IDs to enable.
If no --plugin is provided, load all discovered default_enabled plugins.
```

Example:

```python
def bunnyland_plugins() -> list[Plugin]:
    return [
        Plugin(
            id="my_custom_plugins.foo",
            name="Foo Mechanics",
            ecs=EcsContribution(
                components=(FooComponent,),
                systems=(FooSystem,),
            ),
        ),
        Plugin(
            id="my_custom_plugins.bar",
            name="Bar Mechanics",
            dependencies=("my_custom_plugins.foo",),
            ecs=EcsContribution(
                components=(BarComponent,),
                systems=(BarSystem,),
            ),
        ),
    ]
```

## 25. LLM Agents

### 25.1 LLM controllers

LLM agents are controllers over persistent character entities.

They receive the same prompt foundation as humans, plus tool schemas.

### 25.2 LLM tool surface

Tools should map directly to commands:

```text
move(direction)
take(item)
drop(item)
put(item, container)
use(target, tool?, intent?)
eat(item)
drink(source)
say(text, intent?)
tell(target, text, intent?)
take_note(text, tags?)
remember(query?, mode?, limit?)
write(target, text)
wait()
yield()
```

### 25.3 LLM constraints

LLMs cannot bypass:

```text
Action/Focus costs
controller generation
reachability
containment
policy/boundaries
active/suspended/dead state
validated life-sim prerequisites
typed event/action handlers
```

### 25.4 LLM decision logging

Do not store hidden chain-of-thought.

Store observable/actionable summaries if useful, such as:

```text
Juniper searched her notes because she was unsure whether the basin water was safe.
Hazel chose to warn Juniper because she remembered the water tasted strange.
```

## 26. Persistence

Persist:

```text
Relics world snapshots
executed command history / audit log
typed event log
character notes and memories
vector DB collections
controller state
plugin/world configuration
generated content proposals and approvals
```

Do not persist:

```text

transient Discord prompt leases unless convenient
in-flight LLM futures
engine readiness caches
```

After restart:

```text
world state loads from snapshot/event persistence
command queues are empty
prompts can be regenerated from current ECS state
players/LLMs can submit commands again
```

## 30. Design Rules

1. Characters are persistent; controllers are replaceable.
2. Relics ECS is the authoritative current world state.
3. Commands mutate the world only through the world actor.
4. Queued commands are volatile and do not survive restart.
5. Action and Focus regenerate in real time for all characters.
6. Suspended/no-op controllers do not spend points.
7. Most harmful systems ignore suspended characters via their own queries.
8. Characters can die only while active.
9. LLM-controlled active characters can die.
10. Discord-controlled active characters can die even if the human is idle.
11. Downed state should usually precede death.
12. `Contains` is the only physical containment/possession edge.
13. `Holding` and `Wearing` are active equipment overlays.
14. Hunger and thirst are separate mechanics.
15. Mood is multidimensional affect, not a single scalar.
16. Flirting is speech intent, not a separate base action.
17. say/tell cost both Action and Focus.
18. Notes are private memory and cost Focus.
19. Physical writing is a world action and can be discovered.
20. Room summaries are event-driven projections, not ticking systems.
21. Plugins are extension bundles, not necessarily mechanics.
22. Generators and controllers are plugins but not ECS mechanics.
23. Typed events are preferred.
24. Denied character boundaries always win.
25. Admins cannot override character boundaries.
26. PvP means combat, not dialogue.
27. Pickpocketing is disabled for now.
28. Pregnancy cannot begin while suspended.
29. Pregnancy may progress while suspended, but birth is deferred until active.
30. Exact numbers are tuning data, not architecture.

## 31. Final Summary

bunnyland is an async Discord-first ECS sandbox where persistent characters are controlled by humans, LLM agents, or suspended/no-op controllers. The world is generated and expanded by LLM DM/generator plugins, but all state changes flow through validated commands, typed events, and Relics ECS mutations.

The simulation is designed for emergence. Needs, affect, relationships, environment, inventory, notes, dialogue, social boundaries, and survival pressure all interact. Human and LLM controllers share the same prompts and command tools. Plugins extend the game with life-sim, colony-sim, barbarian-survival, memory, worldgen, Discord, LLM, and custom server behavior.

The foundation is intentionally strict where correctness matters:

```text
single world actor
typed commands
typed events
component replacement
controller generation checks
volatile command queues
Action/Focus cost validation
Contains-only physical containment
policy/boundary gates
system-owned queries
```

That strict core gives bunnyland room to become messy, social, surprising, and alive at the gameplay level without letting the architecture collapse into uncontrolled LLM narration or untestable simulation blobs.
